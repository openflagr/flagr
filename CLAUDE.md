# CLAUDE.md

Guidance for Claude Code sessions working in this repository.

## Overview

`paubox-flagr` is Paubox's fork of [openflagr/flagr](https://github.com/openflagr/flagr),
a Go service that provides feature flags, A/B experimentation, and dynamic
configuration via REST APIs (swagger-generated). The Go module path is
`github.com/paubox/paubox-flagr`.

The service exposes:

- An evaluation API (`/api/v1/evaluation`, `/api/v1/evaluation/batch`) that maps
  an `entityID` + `entityContext` to a flag variant.
- A CRUD API for flags, segments, constraints, distributions, variants, tags.
- An in-memory evaluation cache refreshed from the DB on an interval.
- Optional data-record streams (Kafka, Kinesis, Pub/Sub) for evaluation
  telemetry.
- A bundled web UI under `browser/` (served from the binary).

Paubox runs flagr against MySQL/Aurora behind ProxySQL on Fargate; see
`Dockerfile.proxysql`, `proxysql.cnf.template`, and the `deploy-*` workflows
under `.github/workflows/`.

## Repository Layout

- `pkg/` — application source (the code we own/modify):
  - `pkg/config/` — env-driven configuration (`env.go` is the source of truth
    for `FLAGR_*` variables), middleware setup (auth, CORS, logging, gzip,
    Prometheus, NewRelic, Sentry, Datadog).
  - `pkg/entity/` — GORM models (`flag.go`, `segment.go`, `constraint.go`,
    `distribution.go`, `variant.go`, `tag.go`, `user.go`), DB bootstrap
    (`db.go`), and the bucketing/evaluation core (`evaluator.go`,
    `flag_snapshot.go`).
  - `pkg/handler/` — HTTP handlers wired to the swagger API: `eval.go` and
    `eval_cache*.go` (hot path), `crud*.go` (flag management), `export.go`,
    `validate.go`, and data recorders `data_recorder_{kafka,kinesis,pubsub}.go`.
  - `pkg/mapper/entity_restapi/` — converters between DB entities and REST
    payloads.
  - `pkg/util/` — shared helpers.
- `swagger/` — hand-edited OpenAPI spec sources. `swagger/index.yaml` is the
  entry point; `make api_docs` bundles them into `docs/api_docs/bundle.yaml`.
- `swagger_gen/` — go-swagger-generated server stubs. Regenerated by
  `make swagger`; do not hand-edit (except `swagger_gen/restapi/configure_flagr.go`,
  which is preserved across regenerations and is where the handler chain is
  wired up).
- `docs/` — docsify site (overview, env reference, use cases). Browse via
  `make serve_docs`.
- `integration_tests/` — docker-compose stack and `test.sh` driver for
  end-to-end checks.
- `benchmark/` — vegeta-based load tests for the evaluation endpoint.
- `Dockerfile.go` — runtime image for the flagr server.
- `Dockerfile.proxysql` / `proxysql.cnf.template` — Paubox-specific ProxySQL
  sidecar used in front of Aurora.
- `.github/workflows/` — CI (`ci.yml`, `ci_codeql.yml`) and Paubox deploy
  pipelines (`staging.yml`, `production.yml`, `deploy-app.yml`,
  `deploy-proxysql.yml`).

## Build, Run, Test

Toolchain: Go 1.19 (`CGO_ENABLED=1` is required because of the sqlite3 driver).
The Makefile assumes `go`, `npm`, and `docker` on PATH.

- `make deps` — install pinned `go-swagger@v0.29.0` and
  `golangci-lint@v1.49.0`.
- `make gen` — regenerate `docs/api_docs/bundle.yaml` and the `swagger_gen/`
  server. Run after editing anything under `swagger/`.
- `make build` — compile the server binary to `./flagr` (entrypoint
  `swagger_gen/cmd/flagr-server`).
- `make run` / `make start` — run the server on port 5020.
- `make all` — `deps gen build run` end-to-end.
- `make test` — `go test -race -covermode=atomic` over `pkg/...`, gated by
  `verifiers` (lint + swagger validate).
- `make benchmark` — Go benchmarks in `pkg/...`.
- `make verify_lint` — `golangci-lint run --timeout 5m -D errcheck ./pkg/...`.
- `make verify_swagger_nochange` — regenerates swagger and fails if the diff
  is non-empty; CI relies on this, so commit regenerated `swagger_gen/` when
  swagger changes.
- Integration tests: `cd integration_tests && ./test.sh` (spins up
  docker-compose services).
- Docs preview: `make serve_docs` (uses `docsify-cli`).

## Configuration

All runtime configuration is read from environment variables via
`github.com/caarlos0/env`. The authoritative list lives in
`pkg/config/env.go`. Highlights:

- `HOST`, `PORT` (default `5020`).
- `FLAGR_DB_DBDRIVER` (`sqlite3` | `mysql` | `postgres` | `json_file` |
  `json_http`) and `FLAGR_DB_DBCONNECTIONSTR`. The `json_*` drivers force
  `EvalOnlyMode=true`.
- `FLAGR_EVALCACHE_REFRESHINTERVAL` (default `3s`),
  `FLAGR_EVALCACHE_REFRESHTIMEOUT` (`59s`).
- `FLAGR_LOGRUS_LEVEL`, `FLAGR_LOGRUS_FORMAT` (`text`|`json`).
- `FLAGR_CORS_*` — Paubox defaults include the `X_PBJ_IMP` / `X_PBJ_TOKEN`
  headers used by impersonation flows.
- `FLAGR_BASIC_AUTH_*` and JWT auth knobs (`FLAGR_JWT_AUTH_*`); whitelist
  paths default to `/api/v1/flags` and `/api/v1/evaluation*`.
- Data recorders: `FLAGR_RECORDER_ENABLED`, `FLAGR_RECORDER_TYPE`
  (`kafka`|`kinesis`|`pubsub`), plus type-specific settings.
- Observability: `FLAGR_NEW_RELIC_*`, `FLAGR_SENTRY_*`, `FLAGR_DATADOG_*`,
  `FLAGR_PROMETHEUS_ENABLED`.
- AWS auth for Kinesis follows the standard SDK chain
  (`AWS_ACCESS_KEY_ID`, etc.); Pub/Sub uses
  `FLAGR_RECORDER_PUBSUB_PROJECT_ID` + `FLAGR_RECORDER_PUBSUB_KEYFILE` or
  `GOOGLE_APPLICATION_CREDENTIALS`. See `docs/flagr_env.md` for full
  examples.

## Conventions

- `go.mod` is pinned to `go 1.19`; do not bump without coordinating with the
  Dockerfile (`golang:1.19.0-alpine3.16`) and CI.
- Module import path is `github.com/paubox/paubox-flagr` (not the upstream
  `openflagr/flagr`). Tests reference it as
  `github.com/paubox/paubox-flagr/pkg/...`.
- The fork tracks upstream `openflagr/flagr`. When merging upstream changes,
  preserve Paubox-only files: `Dockerfile.proxysql`,
  `proxysql.cnf.template`, the `deploy-*`/`staging.yml`/`production.yml`
  workflows, and the `CORS` header defaults.
- `swagger_gen/` is generated except for
  `swagger_gen/restapi/configure_flagr.go`, which the Makefile snapshots and
  restores around `swagger generate server`. Edits to handler wiring belong
  there.
- Handler code should go through the swagger-generated principal/operation
  signatures; new endpoints start with a swagger spec change under
  `swagger/`, then `make gen`.
- Tests live next to their sources (`*_test.go`) and use
  `github.com/stretchr/testify` plus `prashantv/gostub` for stubbing.
- `CHANGELOG.md` is auto-generated by `auto-changelog`
  (see `.auto-changelog` and `cd_changelog.yml`); do not edit by hand.
- Releases are tagged `vX.Y.Z` (see `git tag`). Docker images are published
  by `cd_docker.yml`; Paubox deployments are driven by `deploy-app.yml` /
  `deploy-proxysql.yml`.

## Common Tasks

- Adding a flag field: update the GORM model in `pkg/entity/`, the swagger
  schema in `swagger/`, run `make gen`, then adapt
  `pkg/mapper/entity_restapi/` and handlers in `pkg/handler/`.
- Changing evaluation semantics: most logic is in `pkg/entity/evaluator.go`
  and `pkg/handler/eval.go` / `eval_cache.go`. Add benchmarks under
  `benchmark/` for hot-path changes.
- Tweaking ProxySQL: edit `proxysql.cnf.template` and roll via
  `.github/workflows/deploy-proxysql.yml`.
