# refactor: Consolidate binaries under `cmd/`

**Status:** implemented

## Summary

Move the Flagr server's `main.go` from `swagger_gen/cmd/flagr-server/` into `cmd/flagr-server/`, so both binaries (`flagr-server`, `flagr-validate`) live under a single top-level `cmd/` folder. The `make swagger` target generates the stub in `swagger_gen/cmd/` as usual, copies it verbatim to `cmd/flagr-server/main.go`, then deletes `swagger_gen/cmd/`. The file stays auto-generated and in sync with go-swagger's template ‚Äî no hand-maintenance, no drift.

## Problem Frame

- Two binaries live in different places: `cmd/flagr-validate/main.go` (hand-written) and `swagger_gen/cmd/flagr-server/main.go` (go-swagger output, marked `DO NOT EDIT`).
- `swagger_gen/` is a generated tree ‚Äî conceptually wrong place for a hand-maintained entrypoint. The `configure_flagr.go` exception is already handled via /tmp save/restore in the Makefile; adding `main.go` as a second exception doubles the special-casing.
- Build paths reference `github.com/openflagr/flagr/swagger_gen/cmd/flagr-server` in 3 places; consolidating gives a stable, predictable `cmd/<binary>` layout.

**Success criteria:**
1. `cmd/flagr-server/main.go` and `cmd/flagr-validate/main.go` both exist under `cmd/`
2. `swagger_gen/cmd/` does not exist after `make swagger`
3. `make build` produces a working `./flagr` binary from the new path
4. `make test-integration` and `make test-e2e` pass unchanged (they build the server internally)
5. `make swagger` regenerates, copies the stub to `cmd/flagr-server/main.go`, and leaves no `swagger_gen/cmd/` behind
6. Running `make swagger` twice produces no diff in `cmd/flagr-server/main.go` (idempotent copy)

## Key Technical Decisions

1. **Verbatim copy, no modifications.** The generated `main.go` is copied as-is to `cmd/flagr-server/main.go`. No header rewriting, no import changes ‚Äî the file is genuine go-swagger output, and the `// Code generated by go-swagger; DO NOT EDIT.` header is accurate. The imports (`github.com/openflagr/flagr/swagger_gen/restapi`, `.../restapi/operations`) are absolute Go import paths and resolve correctly regardless of where the file lives.

2. **Copy during `make swagger`, not `--exclude-main`.** go-swagger's `--exclude-main` flag (tested and confirmed working) would skip `cmd/` generation entirely, making `cmd/flagr-server/main.go` hand-maintained. We rejected this in favor of the copy approach because the copy keeps the file auto-generated ‚Äî it refreshes automatically whenever go-swagger's template changes, with zero manual reconciliation needed.

3. **`swagger_gen/cmd/` deleted after copy.** Without deletion, `go build ./...` would find two `main` packages for the server (one in `cmd/flagr-server`, one in `swagger_gen/cmd/flagr-server`). The copy-then-delete ensures only the canonical `cmd/flagr-server/` entrypoint exists.

4. **No changes to `swagger_gen/restapi/` or `swagger_gen/models/`.** Only the `swagger_gen/cmd/` subtree is affected. The `configure_flagr.go` /tmp preserve/restore flow stays as-is.

5. **Three build-path call sites updated in lockstep.** All three build the server binary from a relative or absolute path; they must point at `./cmd/flagr-server/` (or the absolute module path `github.com/openflagr/flagr/cmd/flagr-server`). The Dockerfile uses `make build`, so it's covered transitively.

## Scope

**In scope:**
- Create `cmd/flagr-server/main.go` (initial copy from current `swagger_gen/cmd/flagr-server/main.go`)
- Delete `swagger_gen/cmd/` (it will be regenerated and deleted by `make swagger`)
- Update `make swagger` to copy the stub to `cmd/flagr-server/` then delete `swagger_gen/cmd/`
- Update 3 build-path references (Makefile, e2e-server.sh, integration_server_test.go)
- Update AGENTS.md command table

**Out of scope:**
- Merging `flagr-validate` and `flagr-server` into a single multi-subcommand binary (the codified-flags plan explicitly rejected this; go-flags via go-swagger makes subcommands awkward)
- Any change to `swagger_gen/restapi/configure_flagr.go` or its /tmp preservation
- Renaming `swagger_gen/` itself

**Deferred:** none.

## Execution Plan

### Step 1 ‚Äî Create the canonical entrypoint

Copy the current generated `main.go` to its new home:

```bash
mkdir -p cmd/flagr-server
cp swagger_gen/cmd/flagr-server/main.go cmd/flagr-server/main.go
git rm -r swagger_gen/cmd/
```

The file is a verbatim copy ‚Äî no edits. The `// Code generated by go-swagger; DO NOT EDIT.` header stays because the file is generated output. The imports already use absolute paths (`github.com/openflagr/flagr/swagger_gen/restapi`), so they resolve from the new location.

### Step 2 ‚Äî Update build paths (3 sites)

| File | Line | Old | New |
|------|------|-----|-----|
| `Makefile` | 28 | `github.com/openflagr/flagr/swagger_gen/cmd/flagr-server` | `github.com/openflagr/flagr/cmd/flagr-server` |
| `browser/flagr-ui/scripts/e2e-server.sh` | 29 | `./swagger_gen/cmd/flagr-server/` | `./cmd/flagr-server/` |
| `integration_tests/integration_server_test.go` | 138 | `"./swagger_gen/cmd/flagr-server/"` | `"./cmd/flagr-server/"` |

### Step 3 ‚Äî Update `make swagger` to copy + delete

In `Makefile` `swagger` target (lines 102‚Äì109), add two lines after the `configure_flagr.go` restore ‚Äî copy the freshly generated stub to `cmd/`, then delete `swagger_gen/cmd/`:

```make
swagger: verify_swagger
	@echo "Regenerate swagger files"
	@rm -f /tmp/configure_flagr.go
	@cp $(PWD)/swagger_gen/restapi/configure_flagr.go /tmp/configure_flagr.go 2>/dev/null || :
	@rm -rf $(PWD)/swagger_gen
	@mkdir $(PWD)/swagger_gen
	@swagger generate server -t ./swagger_gen -f $(PWD)/docs/api_docs/bundle.yaml
	@cp /tmp/configure_flagr.go $(PWD)/swagger_gen/restapi/configure_flagr.go 2>/dev/null || :
	@mkdir -p $(PWD)/cmd/flagr-server
	@cp $(PWD)/swagger_gen/cmd/flagr-server/main.go $(PWD)/cmd/flagr-server/main.go
	@rm -rf $(PWD)/swagger_gen/cmd
```

Three new lines:
- `mkdir -p cmd/flagr-server` ‚Äî ensures the directory exists (idempotent; safe on first run and subsequent runs)
- `cp swagger_gen/cmd/flagr-server/main.go cmd/flagr-server/main.go` ‚Äî verbatim copy, overwrites any previous copy
- `rm -rf swagger_gen/cmd` ‚Äî removes the generated stub so only `cmd/flagr-server/` holds the entrypoint

### Step 4 ‚Äî Update AGENTS.md

Add a row to the Commands table:

```
| `go build -o flagr ./cmd/flagr-server/` | Build server binary directly (same as `make build`) |
```

(The existing `flagr-validate` row already points at `./cmd/flagr-validate/`.)

### Step 5 ‚Äî Verify

3. `make test-integration` ‚Äî integration tests auto-build and pass
4. `make swagger` ‚Äî regenerates; `git status` shows no `swagger_gen/cmd/` directory, `cmd/flagr-server/main.go` is present and unchanged (idempotent copy)
5. Run `make swagger` a second time ‚Äî `git diff` shows no changes (idempotent)
6. `go build ./cmd/flagr-validate/` ‚Äî validate binary still builds
7. `go vet ./cmd/...` ‚Äî clean

## Risks

- **`DO NOT EDIT` header in a non-generated location.** The file lives in `cmd/flagr-server/` but carries the go-swagger `DO NOT EDIT` header. A developer might be confused about whether they can edit it. Mitigation: the header is accurate ‚Äî the file is generated and overwritten by `make swagger`. Direct edits would be lost on the next regeneration. This is the same contract as every other file in `swagger_gen/`, just in a different directory.
- **Two `main` packages during `make swagger`.** Between the `swagger generate` step and the `rm -rf swagger_gen/cmd` step, both `cmd/flagr-server/main.go` and `swagger_gen/cmd/flagr-server/main.go` exist. This is only a problem if someone runs `go build ./...` mid-`make swagger` ‚Äî not a realistic scenario. The two files are identical at that point anyway.
